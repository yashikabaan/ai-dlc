# Backlog Policy

Last Updated: `<YYYY-MM-DD>`

## Purpose
Define how ideas become executable work while keeping backlog quality high.

## Intake Rules
- Every item must state problem, user impact, and expected outcome.
- Use templates for consistency.
- Items without clear value stay in Idea/Parking status.

## Epic Rules
- Epics represent outcome-oriented chunks, not technical layers.
- Epics must include success signals and constraints.

## User Story Rules
- Stories must be independently valuable and testable.
- Include acceptance criteria and definition of done.
- Split oversized stories before sprint commitment.

## Prioritization Rules
- Primary factors: user value, urgency, risk reduction, dependency timing.
- Tie-breaker rule: `<team rule>`

## Readiness Definition
- [ ] Clear user/value statement
- [ ] Acceptance criteria defined
- [ ] Dependencies identified
- [ ] Risks noted
- [ ] Estimated at working level

## Hygiene Rules
- Review stale items every `<cadence>`.
- Close or archive obsolete work.
- Keep traceability from idea -> epic -> story -> PR.
